🚨 Exception Handling in Java

1. 🎯 Why This Exists (Indian Real-Life Example)

  • In UPI payments, if internet goes off mid-transaction, the app must handle it safely – not double debit your money.
  • In IRCTC, if payment fails after seat is blocked, the system must roll back the booking.
  • In a college result portal, if some marks data is missing, it should show a friendly error, not crash.
  • Similarly, in Java, exceptions let us handle errors gracefully instead of the program suddenly stopping.

2. 📖 What Is an Exception?

  • An exception is an unexpected problem that happens while the program is running (e.g., divide by zero, file not found).
  • In Java, exceptions are objects that describe what went wrong.
  • Without handling them, your program can crash with an error message.

3. 🧾 Syntax / Basic Structure ⭐

Basic try–catch

try {
    // risky code
} catch (ExceptionType e) {
    // handling code
}

Example: Handling divide-by-zero

public class SimpleExceptionDemo {
    public static void main(String[] args) {
        int a = 10;
        int b = 0;

        try {
            int result = a / b;               // risky
            System.out.println(result);
        } catch (ArithmeticException e) {      // handle
            System.out.println("Cannot divide by zero");
        }

        System.out.println("Program continues...");
    }
}

4. 🔍 How It Works (Step-by-Step Execution) ⭐

  1. Normal flow enters try-block – Java starts executing statements inside try { ... }.
  2. Exception occurs – At a / b with b = 0, Java detects an ArithmeticException.
  3. Jump to matching catch-block – Java stops the try-block and looks for catch (ArithmeticException e).
  4. Catch-block runs – Prints "Cannot divide by zero" instead of crashing.
  5. Program continues – The next line after the try–catch prints "Program continues...".

5. ✅ When to Use Exception Handling

  • When dealing with user input (wrong values, empty input).
  • When working with files, database, network – things can go wrong outside your program.
  • Whenever you want to show a clear message instead of a crash.

6. ⛔ When NOT to Use / Misuse

  • Don’t use exceptions for normal flow (e.g., using try–catch instead of simple if checks).
  • Don’t write catch (Exception e) everywhere and ignore the problem.
  • Don’t leave catch blocks empty – always handle or log the error.

7. 🧠 Minimum You MUST Remember

  • try – block where exception might occur.
  • catch – block that handles the exception.
  • finally – (optional) block that always runs (closing files, releasing resources).
  • Two broad types: checked (must handle at compile time) and unchecked (runtime).

8. ⚠️ Common Mistakes

  • Putting too much code inside try – makes debugging harder.
  • Catching generic Exception instead of specific types (like IOException, ArithmeticException).
  • Swallowing exceptions: catch (Exception e) {} with empty body.
  • Not closing resources (files, DB connections) – use finally or try-with-resources.
