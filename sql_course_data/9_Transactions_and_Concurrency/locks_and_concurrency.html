Locks and Concurrency Control in SQL

Concurrency control ensures that multiple database transactions can occur simultaneously without causing data inconsistency. It uses locks to manage access to shared resources.

Types of Locks

  • Shared Lock (S): Allows multiple transactions to read but not modify data.
  • Exclusive Lock (X): Prevents other transactions from reading or writing the data.
  • Intent Locks: Indicate intention to acquire specific locks at a finer level.

Isolation Levels

  • Read Uncommitted: Allows dirty reads.
  • Read Committed: Prevents dirty reads.
  • Repeatable Read: Prevents dirty and non-repeatable reads.
  • Serializable: Ensures full isolation; highest level.

Example

BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

Best Practices

  • Keep transactions short.
  • Access resources in a consistent order.
  • Handle deadlocks by retrying transactions.
