Locks and Concurrency Control in SQL

1️⃣ Why Does This Topic Exist?

Locks and concurrency control help many users work with the same data at the same time without errors. In Indian banks, IRCTC, and e-commerce, this prevents double booking, lost money, or data corruption.

2️⃣ What Are Locks and Concurrency?

Locks are rules that control who can read or write data at a time. Concurrency control means managing many users or programs using the database together.

3️⃣ Syntax/Structure

-- Example: Lock a row for update
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

4️⃣ Step-by-Step Execution

  1. Start a transaction.
  2. Lock the data you want to change (e.g., FOR UPDATE).
  3. Make your changes (UPDATE, DELETE).
  4. Commit or rollback the transaction.

5️⃣ When to Use

  • When many users need to update the same data (e.g., booking tickets, updating balances).
  • To prevent two people from changing the same row at the same time.

6️⃣ When Not to Use

  • For simple SELECT queries that don’t change data.
  • When you don’t need to prevent conflicts (read-only reports).

7️⃣ Minimum to Remember

  • Locks prevent data problems when many users work together.
  • Use locks for updates, not for simple reads.

8️⃣ Common Mistakes

  • Forgetting to release locks (by committing or rolling back).
  • Locking too much data, which slows down the system.
  • Not handling deadlocks (when two users wait for each other).

9️⃣ Interviewer’s Perspective

  • Give a real-life Indian example where locks prevent problems (e.g., IRCTC ticket booking).
  • What is a deadlock? How do you handle it?
