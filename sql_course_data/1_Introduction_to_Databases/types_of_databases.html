🧱 Types of Databases (Relational vs NoSQL)

This topic is to give you a big-picture idea of how different databases are used.

1. 🎯 Why This Exists (Indian Real-Life Example)

  • Banking system:
    • Account number, name, balance, transactions – all highly structured.
    • Data must always be accurate (you can’t “approximate” money).
  • Social media / OTT apps (Instagram, YouTube, Hotstar):
    • Likes, comments, watch history, device info, logs – huge and constantly changing.
    • Some data is less strict; they focus more on speed and scale.
  • Because needs are different, we have different types of databases – mainly Relational and NoSQL.

2. 📖 What Is This

  • Relational Databases (RDBMS): Store data in tables with rows and columns, with relationships between tables.
  • NoSQL Databases: Store data in more flexible forms – JSON documents, key-value pairs, graphs, etc.
  • As a beginner SQL learner, you will mainly work with Relational Databases, but it’s good to know NoSQL exists.

3. 🧾 Syntax / Basic Structure ⭐

a) Relational Database – Table View

Example: Accounts table in a bank (RDBMS like MySQL)

+-----------+----------+---------+
| AccountNo | Name     | Balance |
-----------+----------+---------+
| 1001      | Anjali   | 5000    |
| 1002      | Rahul    | 12000   |
-----------+----------+---------+

Accessed using SQL like:

SELECT Name, Balance
FROM Accounts
WHERE AccountNo = 1002;

b) NoSQL – Document Style (Very High-Level)

Example: Same account stored as a JSON document in a NoSQL DB (like MongoDB):

{
  "AccountNo": 1002,
  "Name": "Rahul",
  "Balance": 12000,
  "City": "Kochi"
}

You don’t need to learn NoSQL syntax now; just see that structure is more flexible.

4. 🔍 How It Works (Step-by-Step Execution) ⭐

A. Relational Database (RDBMS)

  1. Design tables – Decide what tables you need (Students, Courses, Enrollments).
  2. Define columns and types – e.g., StudentID INT, Name VARCHAR(50), Marks INT.
  3. Set relationships – e.g., Enrollments.StudentID refers to Students.StudentID.
  4. Insert data – Add rows into each table.
  5. Use SQL – Run queries like: 
    SELECT s.Name, c.CourseName
FROM Students s
JOIN Enrollments e ON s.StudentID = e.StudentID
JOIN Courses c ON e.CourseID = c.CourseID;
  6. The RDBMS engine ensures rules are followed (no invalid foreign keys, proper data types, etc.).

B. NoSQL (Document / Key-Value)

  1. Instead of many tables, you might keep all related data inside one document.
  2. Structure can vary slightly between documents (flexibility).
  3. Very useful when data shape changes often (e.g., new fields for app tracking).
  4. The system is usually designed for high read/write speed and scaling across servers.

5. ✅ When to Use

Relational Databases (RDBMS)

  • Banking, insurance, railway bookings, billing systems.
  • College ERP, HRMS, hospital records.
  • Any domain where data is highly structured and rules/relationships are important.

NoSQL Databases

  • Social media feeds, comments, likes, chats.
  • Large-scale logging, analytics events, IoT sensor data.
  • Content-heavy apps where each item may have slightly different fields.

6. ⛔ When NOT to Use

Don’t force RDBMS when:

  • Data structure keeps changing every day and is not clearly defined.
  • You mainly store big, unstructured blobs (e.g., raw logs, images metadata) with few relations.

Don’t jump to NoSQL when:

  • You are a beginner and your use case is simple and relational (like student-course-enrollment).
  • You need strong consistency and transactions (e.g., money transfer).

7. 🧠 Minimum You MUST Remember

  • You will mainly work with Relational Databases in an SQL course.
  • Relational DB → tables + rows + columns + relationships; uses SQL.
  • NoSQL DB → more flexible structure (documents, key-value, etc.), used heavily in modern web-scale systems.
  • No single type is “best” – it depends on the problem.

8. ⚠️ Common Mistakes

  • Thinking NoSQL is “better SQL” – it’s just different, not a replacement.
  • Believing you must choose only one forever – many real systems use both (polyglot persistence).
  • Ignoring relational DB basics because NoSQL is “trendy” – interviews still heavily test RDBMS + SQL.

9. 🎤 Interviewer’s Perspective

Common questions:

  • “What is an RDBMS? Name a few examples.”
  • “What is NoSQL? When would you use it?”
  • “Difference between relational and NoSQL databases in simple terms.”
  • “In a banking system vs in a social media app, which type would you prefer and why?”

What they expect:

  • Clear idea that RDBMS uses tables and SQL; NoSQL is more flexible.
  • 1–2 relevant real-life scenarios for each.
  • Not deep theory – just conceptual clarity.
