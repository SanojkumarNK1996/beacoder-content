Transactions and Concurrency

When multiple users are accessing or modifying a database at the same time, it’s important to ensure that the data remains correct, consistent, and reliable. This is handled through transactions and concurrency control in a Database Management System (DBMS).

What is a Transaction?

A transaction is a single unit of work that performs one or more operations (such as insert, update, or delete) on a database. A transaction must follow the ACID properties — Atomicity, Consistency, Isolation, and Durability — to ensure data reliability.

Example: In online banking, transferring ₹500 from Account A to Account B involves:

  1. Deducting ₹500 from Account A
  2. Adding ₹500 to Account B
Both actions together make one transaction. If either fails, the whole transaction is canceled to keep the data consistent.

States of a Transaction

During its execution, a transaction goes through several states:

  • Active: The transaction is currently being executed.
  • Partially Committed: The transaction has finished executing but is not yet permanently saved.
  • Committed: The changes are successfully saved in the database.
  • Failed: An error or crash occurred before completion.
  • Aborted: The transaction is rolled back, undoing any partial changes.

Why Transactions Are Important

  • Ensure data integrity and reliability
  • Allow safe recovery after system failures
  • Prevent partial updates that could corrupt data

What is Concurrency?

Concurrency refers to multiple users or applications accessing the database at the same time. For example, in an e-commerce website, many customers may place orders simultaneously. The DBMS must handle all these requests without conflicts or errors.

Without proper concurrency control, problems like these can occur:

  • Lost Update: When two transactions modify the same data, and one update overwrites the other.
  • Dirty Read: Reading uncommitted changes from another transaction (which might later fail).
  • Inconsistent Read: Getting different results when reading the same data multiple times during another transaction.

Concurrency Control

Concurrency Control ensures that multiple transactions can occur together without interfering with each other, maintaining data accuracy.

Common Techniques:

  • Locking: A transaction locks data while it’s being used so others can’t change it at the same time.
  • Timestamp Ordering: Each transaction gets a timestamp; older transactions get priority to prevent conflicts.
  • Multiversion Concurrency Control (MVCC): Instead of locking, it keeps multiple versions of data so different users can read/write safely at once (used in PostgreSQL, MySQL InnoDB, etc.).

Commit and Rollback

  • COMMIT: Saves all changes made by a transaction permanently.
  • ROLLBACK: Cancels a transaction and restores the database to its previous state.

Example: If a payment process fails halfway, the system performs a ROLLBACK to ensure no partial transaction is left behind.

Summary

Transactions ensure that operations on data happen safely and completely, while concurrency control makes sure that multiple users can access and modify data at the same time without causing errors or inconsistencies.

Together, they help databases handle real-world scenarios like banking, ticket booking, and shopping applications smoothly and reliably.
