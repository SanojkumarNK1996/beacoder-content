💳 Transactions and Concurrency (Real-Life Safe Updates)

1. 🎯 Why This Exists (Indian Real-Life Example)

  • UPI money transfer (PhonePe / GPay):
    • You send ₹500 to a friend.
    • Money must be deducted from your account and added to your friend’s account.
    • Both steps must happen together, even if internet is slow or app crashes.
  • IRCTC ticket booking:
    • Hundreds of people try to book the last few seats at the same time.
    • System must ensure that only one person gets each seat, and others see “WL/Not Available”.
  • Concepts of transactions and concurrency control are what make these operations safe and correct.

2. 📖 What Is This

  • A transaction is a unit of work on the database – usually a group of SQL statements that should be treated as one whole action.
  • Concurrency is when multiple transactions run at the same time.
  • The DBMS uses rules and mechanisms to ensure data stays correct, even under concurrency.

3. 🧾 Syntax / Basic Structure ⭐

In SQL, a typical transaction may look like this (syntax varies slightly by DB):

START TRANSACTION;         -- or BEGIN TRANSACTION

UPDATE Accounts
SET Balance = Balance - 500
WHERE AccountNo = 1001;   -- From account

UPDATE Accounts
SET Balance = Balance + 500
WHERE AccountNo = 2002;   -- To account

COMMIT;                   -- Make all changes permanent
-- ROLLBACK;              -- Use instead of COMMIT to undo

If any statement fails before COMMIT, the DBMS can ROLLBACK to undo all changes in that transaction.

4. 🔍 How It Works (Step-by-Step Execution) ⭐

A. Inside One Transaction

  1. Start
    • DBMS marks a boundary: “Now we’re in one transaction”.
  2. Execute SQL statements
    • Each INSERT, UPDATE, DELETE runs, but changes are not yet permanently written.
  3. Check for errors
    • If any step fails (constraint error, system issue), DBMS can abort the transaction.
  4. COMMIT
    • If everything is fine, COMMIT is executed.
    • Now all changes become permanent (Durability).
  5. ROLLBACK (if needed)
    • If there’s a problem, DBMS uses ROLLBACK to undo all changes since the transaction started.

B. When Many Transactions Run Together (Concurrency)

  • DBMS may lock rows or use other techniques so that two people don’t update the same record wrongly.
  • It schedules operations so that final result is as if each transaction ran one after another (Isolation).
  • It prevents problems like:
    • Lost Update – one person’s update silently overwriting another’s.
    • Dirty Read – reading uncommitted, temporary changes from another transaction.
    • Inconsistent Read – getting half-old, half-new data while another transaction is in progress.

5. ✅ When to Use Transactions

  • Whenever multiple changes must be treated as one logical operation (money transfer, seat booking, order + payment).
  • When failures (power cut, app crash) should not leave partial data.
  • When correctness is more important than speed.

6. ⛔ When NOT to Use (or Keep Very Small)

  • Don’t keep a transaction open for too long (e.g., waiting for user input), as it may hold locks and block others.
  • Don’t put unrelated operations in the same transaction just to “group” them; it can reduce concurrency.

7. 🧠 Minimum You MUST Remember

  • A transaction = unit of work that should be all-or-nothing.
  • COMMIT makes changes permanent; ROLLBACK undoes them.
  • Concurrency = many transactions running at once; DBMS keeps data consistent using locks or similar techniques.
  • Common issues: lost update, dirty read, inconsistent read.

8. ⚠️ Common Mistakes

  • Forgetting to use transactions around multiple related updates (e.g., only updating one side of a transfer).
  • Leaving transactions open (no commit/rollback) – can lock rows and slow down the system.
  • Assuming reads never interfere – some isolation levels can still show intermediate states if not configured properly (advanced topic).

9. 🎤 Interviewer’s Perspective

Common questions:

  • “What is a transaction? Give a real-life example.”
  • “What are COMMIT and ROLLBACK?”
  • “What is concurrency? Why is it challenging in databases?”
  • “Explain lost update / dirty read / inconsistent read with an example.”
  • “How do transactions relate to ACID properties?”

What they expect:

  • Clear UPI / banking / ticket-booking stories to explain transactions.
  • Basic understanding of why concurrency control is needed.

10. 📝 Practice Questions ⭐

  1. Very Easy – Transaction Story:

    Write a 4–5 step story of a transaction in any app you use (UPI, Swiggy, IRCTC).

  2. Easy – COMMIT vs ROLLBACK:

    Explain the difference between COMMIT and ROLLBACK with a small example.

  3. Easy – Identify Issues:

    Give one example each for: lost update and dirty read in a banking or shopping scenario.

  4. Medium – Write SQL:

    Write SQL statements to transfer ₹1000 from account 101 to account 202 using a transaction (use START TRANSACTION, UPDATE, COMMIT).

  5. Medium – Concurrency Thinking:

    Explain what could go wrong if two people try to update the same row at the same time and there is no concurrency control.
