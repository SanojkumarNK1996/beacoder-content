ACID Properties (Atomicity, Consistency, Isolation, Durability)

When working with databases, it’s very important to make sure that data remains correct, reliable, and safe — even if something goes wrong, like a power failure or multiple people trying to update the same record at once. To guarantee this, databases follow four key rules called the ACID properties.

ACID stands for:

  • Atomicity
  • Consistency
  • Isolation
  • Durability

Let’s understand each of these in simple terms.

1. Atomicity – “All or Nothing”

Atomicity means that a transaction (a group of actions) must be completed entirely or not at all. If any part of the transaction fails, the entire operation is undone.

Example: Imagine transferring ₹1,000 from Account A to Account B. This involves two actions:

  1. Deduct ₹1,000 from Account A
  2. Add ₹1,000 to Account B
If the system crashes after the deduction but before adding to Account B, Atomicity ensures both actions are rolled back, so no money is lost or duplicated.

In short: Either all parts of a transaction succeed, or none do.

2. Consistency – “Data Must Be Correct”

Consistency ensures that data always follows the rules defined in the database. After any transaction, the database must move from one valid state to another valid state.

Example: In a banking database, an account balance cannot be negative. If a transaction would make it negative, it is rejected to keep the data consistent.

  • Prevents invalid or corrupt data from being saved
  • Maintains logical correctness of data

3. Isolation – “Transactions Should Not Interfere”

Isolation means that when multiple users perform transactions at the same time, they don’t affect each other’s results. Each transaction acts as if it’s happening alone.

Example: Suppose two users try to buy the last available movie ticket at once. The database ensures that only one transaction succeeds — not both — even though both requests arrived at nearly the same time.

This prevents problems like:

  • Dirty reads (reading uncommitted data)
  • Lost updates (one transaction overwriting another)
  • Inconsistent reads (seeing partial changes)

4. Durability – “Changes Stay Saved”

Durability guarantees that once a transaction is successfully completed, its changes are permanently saved — even if there’s a system crash or power failure afterward.

Example: If you complete a bank transfer and see the confirmation message, the data is safely written to storage. Even if the computer shuts down right after, your transaction remains recorded.

  • Ensures permanent data storage
  • Uses backups, logs, and recovery systems

Summary

The ACID properties ensure that databases handle data safely and reliably, especially in multi-user and high-volume environments:

Property Meaning Guarantee
Atomicity All actions in a transaction happen completely or not at all No partial updates
Consistency Data remains valid and follows all rules No invalid data states
Isolation Simultaneous transactions do not interfere Accurate results under concurrency
Durability Once committed, data changes stay saved Survives crashes or failures

Together, these four properties make databases dependable and trustworthy — ensuring data accuracy and stability even under unexpected conditions.
