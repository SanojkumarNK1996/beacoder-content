🔑 Keys in Databases (Primary, Foreign, Unique, Composite)

1. 🎯 Why This Exists (Indian Real-Life Example)

  • School / College:
    • There can be many students named “Rahul Kumar”.
    • The college uses a unique roll number to identify each student. That roll number works like a Primary Key.
  • Swiggy / Zomato order:
    • Each order has a unique Order ID (primary key).
    • That Order ID is linked to a customer, restaurant, and delivery partner using keys.
  • Without keys, the database cannot clearly say which record is which or how different tables are related.

2. 📖 What Is This

  • A key is one column (or a group of columns) used to uniquely identify a row or to link tables.
  • Keys help avoid duplicates, maintain relationships, and keep data accurate.
  • Common types: Primary Key, Foreign Key, Unique Key, Composite Key (and others in advanced topics).

3. 🧾 Syntax / Basic Structure ⭐

We’ll see how keys look when defining a table using SQL (simplified):

a) Primary Key

CREATE TABLE Students (
  Student_ID INT PRIMARY KEY,
  Name       VARCHAR(50),
  Class      VARCHAR(10)
);

b) Foreign Key

CREATE TABLE Marks (
  Student_ID INT,
  Marks      INT,
  CONSTRAINT fk_student
    FOREIGN KEY (Student_ID) REFERENCES Students(Student_ID)
);

c) Unique Key

CREATE TABLE Employees (
  Employee_ID INT PRIMARY KEY,
  Email       VARCHAR(100) UNIQUE
);

d) Composite Key

CREATE TABLE Course_Enrollment (
  Student_ID INT,
  Course_ID  VARCHAR(10),
  PRIMARY KEY (Student_ID, Course_ID)
);

4. 🔍 How It Works (Step-by-Step Execution) ⭐

Example: Students and Marks

Step 1 – Define Students Table with Primary Key

CREATE TABLE Students (
  Student_ID INT PRIMARY KEY,
  Name       VARCHAR(50),
  Class      VARCHAR(10)
);
  • Student_ID is marked as PRIMARY KEY.
  • DBMS will reject any attempt to insert a row with a duplicate Student_ID or NULL.

Step 2 – Insert Students

INSERT INTO Students (Student_ID, Name, Class)
VALUES (101, 'Rahul', '10A');

INSERT INTO Students (Student_ID, Name, Class)
VALUES (101, 'Asha', '10B');  -- ❌ This will fail (duplicate PK)
  • Second insert fails because 101 already exists – primary key protects uniqueness.

Step 3 – Define Marks Table with Foreign Key

CREATE TABLE Marks (
  Student_ID INT,
  Marks      INT,
  CONSTRAINT fk_student
    FOREIGN KEY (Student_ID) REFERENCES Students(Student_ID)
);
  • Now Marks.Student_ID must refer to an existing Students.Student_ID.

Step 4 – Insert Marks

INSERT INTO Marks (Student_ID, Marks)
VALUES (101, 85);   -- ✅ OK (Student 101 exists)

INSERT INTO Marks (Student_ID, Marks)
VALUES (999, 90);   -- ❌ Fails (no such student)
  • Foreign key ensures you can’t store marks for a non-existent student.

5. ✅ When to Use Keys

  • Primary Key – Always choose one for every table to uniquely identify each row.
  • Foreign Key – Whenever you want to link two tables (Student–Marks, Customer–Orders).
  • Unique Key – For columns that must be unique (email, Aadhaar, PAN, phone in some systems).
  • Composite Key – When combination of columns is unique (Student+Course, Account+IFSC).

6. ⛔ When NOT to Use / Be Careful

  • Don’t randomly mark many columns as primary key; there should be only one primary key (though it can be composite).
  • Don’t create foreign keys if data is not truly related – it will restrict inserts/deletes unnecessarily.
  • Too many constraints without understanding can make inserts/updates harder during early prototyping.

7. 🧠 Minimum You MUST Remember

  • Primary Key – uniquely identifies each row, no NULL, no duplicates.
  • Foreign Key – refers to primary key of another table; enforces relationship/integrity.
  • Unique Key – values must be unique, but can usually be NULL (depends on DB).
  • Composite Key – more than one column together act as a primary key.

8. ⚠️ Common Mistakes

  • Not defining any primary key and later struggling with duplicate rows.
  • Using a changing field (like Name or Phone) as primary key instead of a stable ID.
  • Deleting a row from parent table (Students) without thinking about related child rows (Marks) – can cause foreign key errors.
  • Assuming foreign key automatically creates index (depends on DB; often you should index it yourself).

9. 🎤 Interviewer’s Perspective

Common fresher questions:

  • “What is a primary key? Give an example.”
  • “What is a foreign key and why is it used?”
  • “Difference between primary key and unique key?”
  • “What is a composite key? When would you use it?”
  • “How do keys help in maintaining referential integrity?”

What they expect:

  • Understanding that keys help avoid duplicates and link tables correctly.

10. 📝 Practice Questions ⭐

  1. Very Easy – Identify Keys:

    For a Students table with (RollNo, Name, Class, Phone), which column would you choose as primary key and why?

  2. Easy – Design Keys:

    Design simple tables for Customers and Orders. Decide primary keys for each, and choose a foreign key that connects them.

  3. Easy – Unique vs Primary:

    Explain with an example how primary key and unique key are similar and how they differ.

  4. Medium – Composite Key Scenario:

    For a Train_Booking table with (TrainNo, Date, SeatNo, PassengerID), what could be a good composite key? Explain your choice.

  5. Medium – Write SQL:

    Write a CREATE TABLE statement for a Library_Issues table where (Student_ID, Book_ID) together form the primary key.
