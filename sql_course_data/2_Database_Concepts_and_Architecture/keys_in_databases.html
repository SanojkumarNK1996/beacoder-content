Keys in Databases

In a database, a key is a field (or a combination of fields) that helps identify each record in a table uniquely. Keys are crucial for organizing data, ensuring accuracy, and establishing relationships between tables.

Without keys, the database would not know how to differentiate between two records that look similar — just like a school wouldn’t know which “Rahul Kumar” is which student without a roll number!

Why Are Keys Important?

  • They uniquely identify each record in a table.
  • They help establish relationships between tables.
  • They prevent duplicate or inconsistent data.
  • They make searching, sorting, and linking data faster and more efficient.

Types of Keys in Databases

1. Primary Key

A Primary Key uniquely identifies each record in a table. No two rows can have the same primary key, and it cannot be empty (NULL).

Example: In a Students table, the Student_ID can be the primary key because each student has a unique ID.

Students Table
+-------------+-----------+--------+
| Student_ID  | Name      | Class  |
+-------------+-----------+--------+
| 101         | Rahul     | 10A    |
| 102         | Asha      | 10A    |
| 103         | Kiran     | 9B     |
+-------------+-----------+--------+

Key point: Each Student_ID is unique — no duplicates allowed.

2. Foreign Key

A Foreign Key is used to link two tables together. It refers to the Primary Key in another table, creating a relationship between the two.

Example: The Student_ID in the Marks table can be a foreign key that connects to the Student_ID in the Students table.

Marks Table
+-------------+----------+
| Student_ID  | Marks    |
+-------------+----------+
| 101         | 85       |
| 102         | 90       |
| 103         | 78       |
+-------------+----------+

Now, both tables are related by the Student_ID field.

Key point: Foreign keys maintain data integrity — you can’t have marks for a student who doesn’t exist in the Students table.

3. Unique Key

A Unique Key ensures that all values in a column are different, just like a primary key. However, unlike the primary key, a table can have multiple unique keys, and they can contain NULL values (in some databases).

Example: In an Employees table, both Employee_ID and Email can be unique, since no two employees can share the same email address.

Employees Table
+--------------+----------------------+
| Employee_ID  | Email                |
+--------------+----------------------+
| 201          | aisha@company.com    |
| 202          | rahul@company.com    |
| 203          | kiran@company.com    |
+--------------+----------------------+

4. Composite Key

A Composite Key is created by combining two or more columns to uniquely identify a record when one column alone is not enough.

Example: In a Course_Enrollment table, neither Student_ID nor Course_ID is unique by itself, but the combination of both uniquely identifies each enrollment.

Course_Enrollment Table
+-------------+------------+
| Student_ID  | Course_ID  |
+-------------+------------+
| 101         | C101       |
| 101         | C102       |
| 102         | C101       |
+-------------+------------+

Key point: The pair (Student_ID, Course_ID) is unique for each record.

Summary

Key Type Purpose Can be NULL? Can have duplicates?
Primary Key Uniquely identifies each record No No
Foreign Key Creates a link between tables Yes Yes (if referencing table allows)
Unique Key Ensures all values are unique Yes No
Composite Key Combination of columns used for unique identification No (if any column is part of the key) No (as a pair/group)

Keys play a critical role in organizing and linking data efficiently. They ensure accuracy, consistency, and proper relationships between tables — the foundation of a well-structured database.
