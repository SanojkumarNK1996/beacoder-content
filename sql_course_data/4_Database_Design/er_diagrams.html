ER Diagrams (Entity–Relationship Modeling)

When we create a database, we must first understand what kind of data we’ll store and how those pieces of data are connected. This planning step is called Data Modeling.

The most common way to do this is by drawing an ER Diagram (Entity–Relationship Diagram). It’s like a blueprint of your database — it shows what data you have and how different parts are related.

What is an Entity?

An Entity is something you want to store information about — a person, place, object, or event.

Examples:

  • Student — has Name, Age, and Roll Number
  • Course — has Course Name and Duration
  • Teacher — has Name and Subject

In an ER diagram, an Entity is shown as a rectangle.

+-----------+
|  Student  |
+-----------+

What is an Attribute?

Each Entity has Attributes — the specific details you store about it.

Example:

For the Student entity, attributes could be:

  • StudentID
  • Name
  • Age
  • Email

In an ER diagram, Attributes are shown as ovals connected to their Entity.

          +-----------+
          |  Student  |
          +-----------+
          /    |    \
      Name   Age   Email

What is a Relationship?

A Relationship describes how two entities are connected.

Example:

A Student enrolls in a Course.

In an ER diagram, a Relationship is shown as a diamond shape connecting entities.

+-----------+     enrolls in     +-----------+
|  Student  |--------------------|  Course   |
+-----------+                    +-----------+

Types of Relationships

There are three main types of relationships in database design:

1. One-to-One (1:1)

Each record in one entity is connected to only one record in another entity.

Example:

Each Student has exactly one Library Card.

+-----------+      1:1      +---------------+
|  Student  |----------------| Library Card |
+-----------+                +---------------+

2. One-to-Many (1:N)

One record in an entity can be linked to many records in another entity.

Example:

One Teacher can teach many Courses.

+-----------+      1:N      +-----------+
|  Teacher  |----------------|  Course   |
+-----------+                +-----------+

But each course is taught by only one teacher.

3. Many-to-Many (M:N)

Multiple records in one entity can relate to multiple records in another entity.

Example:

Many Students can enroll in many Courses.

+-----------+      M:N      +-----------+
|  Student  |----------------|  Course   |
+-----------+                +-----------+

In databases, we handle this by creating a junction (link) table — for example, an Enrollment table:

+-----------+-----------+
| StudentID | CourseID  |
+-----------+-----------+
| 1         | C101      |
| 1         | C102      |
| 2         | C101      |
+-----------+-----------+

Keys in ER Diagrams

Keys help identify unique records and connect different tables.

  • Primary Key: A unique identifier for each record (e.g., StudentID).
  • Foreign Key: A column that refers to the Primary Key of another table (e.g., CourseID in Enrollment table).

Example with Keys:

Students Table
+-----------+-----------+------+
| StudentID | Name      | Age  |
+-----------+-----------+------+
| 1         | Riya      | 21   |
| 2         | Arjun     | 22   |
+-----------+-----------+------+

Courses Table
+-----------+----------------+
| CourseID  | CourseName     |
+-----------+----------------+
| C101      | Math           |
| C102      | English        |
+-----------+----------------+

Enrollment Table
+-----------+-----------+
| StudentID | CourseID  |
+-----------+-----------+
| 1         | C101      |
| 1         | C102      |
| 2         | C101      |
+-----------+-----------+

Cardinality

Cardinality defines how many instances of one entity can be associated with another.

Type Meaning Example
One-to-One One entity instance linked to one other Student ↔ LibraryCard
One-to-Many One entity instance linked to many others Teacher ↔ Courses
Many-to-Many Many entity instances linked to many others Students ↔ Courses

Example: Full ER Diagram (Text Version)

+-----------+         +-------------+          +-----------+
|  Student  |         | Enrollment  |          |  Course   |
+-----------+         +-------------+          +-----------+
| StudentID |<------->| StudentID   |<-------->| CourseID  |
| Name      |         | CourseID    |          | Name      |
| Age       |         +-------------+          | Duration  |
+-----------+                                      +-----------+

This ER model shows that:

  • Each student can enroll in many courses.
  • Each course can have many students.
  • The Enrollment table links them together.

Real-Life Analogy

Think of ER modeling as planning a wedding:

  • Entities: Guests, Tables, Dishes
  • Attributes: Guest Name, Table Number, Dish Name
  • Relationships: Guests sit at tables; tables serve dishes.

Just as you plan who sits where and who eats what before the wedding, you plan your data connections in an ER diagram before building the database!

Summary

Concept Symbol Description
Entity Rectangle Represents an object or concept (like Student or Course)
Attribute Oval Represents properties of an entity
Relationship Diamond Shows how entities are related
Primary Key Underlined attribute Unique identifier for an entity

In short: ER diagrams help you plan your database clearly, just like a map helps you plan a journey.
