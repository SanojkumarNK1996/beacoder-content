Indexing Basics

As your database grows, it may store thousands or even millions of records. Finding a specific record in such a large table can take time if the database has to check every single row.

This is where Indexing comes in. An Index makes data search faster — just like an index in the back of a book helps you quickly find a topic without reading every page.

What is an Index?

An Index is a special data structure that stores a small portion of your data in an organized way (usually sorted), so the database can locate information faster.

Example (Book Analogy):

Imagine a 500-page book. You want to find where “Database Normalization” is mentioned.

  • Without an index — you’d flip through every page (slow).
  • With an index — you look at the back of the book, find “Database Normalization → page 120” (fast).

The same idea applies in databases. Instead of checking every record, the database uses an index to jump directly to the location where the data exists.

Why Indexing is Important

Indexes improve the speed of data retrieval (SELECT queries) dramatically, especially in large databases.

However, indexes also take up extra storage space and can slightly slow down data modifications (INSERT, UPDATE, DELETE), because the index needs to be updated whenever data changes.

In short:

  • ✅ Speeds up searches
  • ❌ Slightly slows down data updates
  • ✅ Great for frequently searched columns

How Indexing Works (Simple Example)

Let’s say we have a table of students:

Students Table
+-----------+-----------+--------+
| StudentID | Name      | Age    |
+-----------+-----------+--------+
| 1         | Riya      | 21     |
| 2         | Arjun     | 22     |
| 3         | Meera     | 20     |
| 4         | Ravi      | 23     |
+-----------+-----------+--------+

Without an index, if you run this query:

SELECT * FROM Students WHERE Name = 'Meera';

The database has to check every row until it finds “Meera.” This process is called a full table scan.

If we create an index on the Name column:

CREATE INDEX idx_name ON Students(Name);

The database now stores a separate, sorted list of names — something like this:

Index on Name
+-----------+-----------+
| Name      | StudentID |
+-----------+-----------+
| Arjun     | 2         |
| Meera     | 3         |
| Ravi      | 4         |
| Riya      | 1         |
+-----------+-----------+

Now, when you search for Name = 'Meera', the database looks in this small, sorted list — finds Meera immediately — and jumps to the corresponding record using StudentID = 3.

Types of Indexes

There are several types of indexes, depending on how they are used and stored:

1️⃣ Primary Index

Automatically created on the Primary Key column. It ensures each record is unique and helps in fast retrieval.

CREATE TABLE Students (
  StudentID INT PRIMARY KEY,
  Name VARCHAR(50),
  Age INT
);

Here, a primary index is automatically created on StudentID.

2️⃣ Unique Index

Ensures that no two rows have the same value in the indexed column (useful for things like email or username).

CREATE UNIQUE INDEX idx_email ON Users(Email);

3️⃣ Composite Index

An index on more than one column — used when queries often filter by multiple fields together.

CREATE INDEX idx_name_age ON Students(Name, Age);

This helps when queries use both columns:

SELECT * FROM Students WHERE Name = 'Riya' AND Age = 21;

4️⃣ Full-Text Index

Used for searching within text fields (like searching keywords in articles or documents).

CREATE FULLTEXT INDEX idx_content ON Articles(Content);

How Indexes Are Stored

Indexes are usually stored using special data structures like:

  • B-Tree (Balanced Tree): The most common — helps with range-based lookups (like finding all ages between 20 and 25).
  • Hash Index: Great for exact lookups (like finding a record with a specific ID).

You don’t need to manage these structures yourself — the database handles it internally.

When to Use Indexes

  • When the column is frequently used in WHERE conditions.
  • When the column appears often in JOIN operations.
  • When the column is used in ORDER BY or GROUP BY.

But avoid creating too many indexes — each one adds storage cost and slows down updates slightly.

Real-Life Analogy

Think of a supermarket:

  • Without indexes: you walk every aisle looking for “pasta” (slow).
  • With indexes: signs tell you “Aisle 4 – Pasta” (fast).

The signs (indexes) make searching faster, but every time the store adds a new item, someone must update the sign — just like how indexes need to be updated when data changes.

Summary

Concept Description Example
Index Structure that speeds up searches Index on Name column
Primary Index Automatically created on the Primary Key PRIMARY KEY (StudentID)
Unique Index Ensures no duplicate values INDEX ON Email

In short: An index is like a shortcut — it helps the database find information quickly, just like a book index or store sign helps you find what you need faster.
