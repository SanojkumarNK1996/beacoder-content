Database Normalization

When designing a database, our goal is to organize data so that it is easy to manage, efficient, and free from duplication or errors. This process of organizing data properly is called Database Normalization.

Normalization helps in reducing data redundancy (repeated data) and ensures that the data is stored logically and efficiently.

Why Do We Need Normalization?

Imagine a simple table called Students:

+-----------+-----------+-------------------+
| StudentID | Name      | Courses           |
+-----------+-----------+-------------------+
| 1         | Riya      | Math, English     |
| 2         | Arjun     | Science, Math     |
| 3         | Meera     | English           |
+-----------+-----------+-------------------+

Here, the Courses column has multiple values in a single cell, making it hard to search or update. If a student adds or removes a course, it’s difficult to manage this data correctly.

Normalization fixes such problems by breaking data into smaller, related tables — ensuring each piece of data is stored only once and correctly linked.

Steps (Normal Forms) in Database Normalization

Normalization happens step-by-step, each step known as a Normal Form. Let’s go through them one by one with examples.

1. First Normal Form (1NF)

A table is in 1NF if:

  • Each column has atomic (indivisible) values.
  • Each record is unique (no duplicate rows).

In simple words, each cell should have only one value, not a list.

From our earlier example, the “Courses” column violates 1NF because it contains multiple values.

Converting to 1NF:

+-----------+-----------+---------+
| StudentID | Name      | Course  |
+-----------+-----------+---------+
| 1         | Riya      | Math    |
| 1         | Riya      | English |
| 2         | Arjun     | Science |
| 2         | Arjun     | Math    |
| 3         | Meera     | English |
+-----------+-----------+---------+

Now, every cell has one value — the table is in 1NF.

Analogy: Think of a bookshelf where each slot holds only one book, not a pile of books together — easy to find and organize!

2. Second Normal Form (2NF)

A table is in 2NF if:

  • It is already in 1NF, and
  • Every non-key column depends on the entire primary key, not just part of it.

2NF mainly applies when you have a composite key (a key made of two or more columns).

Example:

In our 1NF table, the combination of (StudentID + Course) could be the primary key.

If we add more columns, say “StudentPhone” and “CourseDuration”, we notice:

  • “StudentPhone” depends only on StudentID
  • “CourseDuration” depends only on Course

So, we split the data into two tables to remove partial dependency:

Students Table:

+-----------+-----------+--------------+
| StudentID | Name      | Phone        |
+-----------+-----------+--------------+
| 1         | Riya      | 9876543210   |
| 2         | Arjun     | 8765432109   |
| 3         | Meera     | 7654321098   |
+-----------+-----------+--------------+

Courses Table:

+---------+---------------+
| Course  | Duration      |
+---------+---------------+
| Math    | 3 Months      |
| English | 2 Months      |
| Science | 4 Months      |
+---------+---------------+

Enrollment Table (linking both):

+-----------+---------+
| StudentID | Course  |
+-----------+---------+
| 1         | Math    |
| 1         | English |
| 2         | Science |
| 2         | Math    |
| 3         | English |
+-----------+---------+

Now, each table focuses on one subject — students, courses, and their relationship.

3. Third Normal Form (3NF)

A table is in 3NF if:

  • It is in 2NF, and
  • There are no transitive dependencies (no non-key column depends on another non-key column).

Example:

Suppose in the Students table, we have:

+-----------+-----------+--------------+------------+
| StudentID | Name      | City         | Pincode    |
+-----------+-----------+--------------+------------+
| 1         | Riya      | Delhi        | 110001     |
| 2         | Arjun     | Mumbai       | 400001     |
| 3         | Meera     | Delhi        | 110001     |
+-----------+-----------+--------------+------------+

Here, “City” and “Pincode” are related — knowing the Pincode tells us the City. So, “City” depends on another non-key column “Pincode”.

This violates 3NF. To fix it, we create a new table for city details.

City Table:

+------------+--------------+
| Pincode    | City         |
+------------+--------------+
| 110001     | Delhi        |
| 400001     | Mumbai       |
+------------+--------------+

Now the Students table becomes:

+-----------+-----------+------------+
| StudentID | Name      | Pincode    |
+-----------+-----------+------------+
| 1         | Riya      | 110001     |
| 2         | Arjun     | 400001     |
| 3         | Meera     | 110001     |
+-----------+-----------+------------+

Now all non-key columns depend only on the key — the table is in 3NF.

4. Boyce-Codd Normal Form (BCNF)

BCNF is an advanced version of 3NF. It removes certain anomalies that 3NF doesn’t handle.

A table is in BCNF if:

  • It is in 3NF, and
  • For every dependency (A → B), A should be a candidate key.

Example:

Suppose we have a table that stores which teacher teaches which subject in which classroom:

+-----------+-----------+-----------+
| Teacher   | Subject   | Classroom |
+-----------+-----------+-----------+
| Meena     | Math      | 101       |
| Arjun     | Science   | 102       |
| Meena     | English   | 101       |
+-----------+-----------+-----------+

Here, a classroom can have only one teacher, but a teacher can teach multiple subjects. So, “Classroom → Teacher” (classroom determines teacher), but Teacher is not a candidate key.

To bring this to BCNF, we split the table:

Classroom Table:

+-----------+-----------+
| Classroom | Teacher   |
+-----------+-----------+
| 101       | Meena     |
| 102       | Arjun     |
+-----------+-----------+

Teaching Table:

+-----------+-----------+
| Teacher   | Subject   |
+-----------+-----------+
| Meena     | Math      |
| Meena     | English   |
| Arjun     | Science   |
+-----------+-----------+

Now both tables are properly structured with no anomalies.

Summary of Normal Forms

Normal Form Main Rule Purpose
1NF Each cell has a single value Remove repeating groups or lists
2NF No partial dependency Ensure all columns depend on the whole key
3NF No transitive dependency Remove indirect dependencies between non-key columns
BCNF Every determinant must be a candidate key Stronger version of 3NF — removes all anomalies

Real-Life Analogy

Think of normalization like organizing your wardrobe:

  • 1NF: Don’t mix different types of clothes in one drawer.
  • 2NF: Separate clothes by category — shirts, pants, etc.
  • 3NF: Don’t store unrelated items (like books) in your wardrobe.
  • BCNF: Each drawer is used only for one type of item and no confusion remains.

Normalization makes your database “neat and tidy,” just like an organized wardrobe!
