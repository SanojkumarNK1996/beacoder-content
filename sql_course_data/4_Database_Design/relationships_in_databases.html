Relationships in Databases

In real life, data is connected — students join courses, teachers teach subjects, customers place orders, and so on. A database must capture these connections to make sense of the data. These connections are called Relationships.

Understanding relationships helps you organize your data correctly, avoid duplication, and make your database more powerful.

What is a Relationship?

A Relationship defines how two or more tables in a database are connected using keys. Usually, this is done through a Primary Key in one table and a Foreign Key in another.

Example:

Let’s say we have two tables:

Students Table
+-----------+-----------+
| StudentID | Name      |
+-----------+-----------+
| 1         | Riya      |
| 2         | Arjun     |
+-----------+-----------+

Courses Table
+-----------+--------------+
| CourseID  | CourseName   |
+-----------+--------------+
| 101       | Math         |
| 102       | English      |
+-----------+--------------+

Now, we want to connect which student takes which course. That connection forms a relationship.

Types of Relationships

There are three main types of relationships in databases:

  1. One-to-One (1:1)
  2. One-to-Many (1:N)
  3. Many-to-Many (M:N)
---

1️⃣ One-to-One (1:1) Relationship

Each record in one table is related to only one record in another table.

Example:

Each student has exactly one student ID card, and each ID card belongs to one student.

Students Table
+-----------+-----------+
| StudentID | Name      |
+-----------+-----------+
| 1         | Riya      |
| 2         | Arjun     |
+-----------+-----------+

StudentCards Table
+-----------+-------------+
| CardID    | StudentID   |
+-----------+-------------+
| 1001      | 1           |
| 1002      | 2           |
+-----------+-------------+

Here, StudentID is the link between both tables.

How it works:

  • One student → one card
  • One card → one student

In this case, StudentID in the StudentCards table is a Foreign Key referencing the Students table.

2️⃣ One-to-Many (1:N) Relationship

This is the most common relationship in databases. One record in one table can be linked to many records in another table.

Example:

One teacher teaches many courses, but each course is taught by only one teacher.

Teachers Table
+-----------+-----------+
| TeacherID | Name      |
+-----------+-----------+
| 1         | Ms. Anu   |
| 2         | Mr. Ravi  |
+-----------+-----------+

Courses Table
+-----------+---------------+------------+
| CourseID  | CourseName    | TeacherID  |
+-----------+---------------+------------+
| 101       | Math          | 1          |
| 102       | English       | 1          |
| 103       | Science       | 2          |
+-----------+---------------+------------+

Here, TeacherID is a Foreign Key in the Courses table that links each course to the teacher who teaches it.

How it works:

  • One teacher → many courses
  • One course → one teacher

3️⃣ Many-to-Many (M:N) Relationship

In this type, many records in one table can be related to many records in another table. This is common in situations like students enrolling in courses — one student can take many courses, and each course can have many students.

Example:

We need a third table (called a junction table or linking table) to connect them.

Students Table
+-----------+-----------+
| StudentID | Name      |
+-----------+-----------+
| 1         | Riya      |
| 2         | Arjun     |
+-----------+-----------+

Courses Table
+-----------+---------------+
| CourseID  | CourseName    |
+-----------+---------------+
| 101       | Math          |
| 102       | English       |
+-----------+---------------+

Enrollments Table (Junction Table)
+-----------+-----------+
| StudentID | CourseID  |
+-----------+-----------+
| 1         | 101       |
| 1         | 102       |
| 2         | 101       |
+-----------+-----------+

This Enrollments table creates a link between Students and Courses.

How it works:

  • One student → many courses
  • One course → many students

Without the junction table, you can’t represent this type of relationship properly in a relational database.

Relationship Summary

Type Meaning Example
One-to-One (1:1) Each record in one table matches exactly one record in another. Student ↔ StudentCard
One-to-Many (1:N) One record in one table can match many records in another. Teacher ↔ Courses
Many-to-Many (M:N) Many records in one table can match many in another (via junction table). Students ↔ Courses

Real-Life Analogies

  • One-to-One: Each person has one passport.
  • One-to-Many: A mother has many children.
  • Many-to-Many: Actors work in many movies; each movie has many actors.

Why Relationships Are Important

Relationships help:

  • Reduce data duplication
  • Ensure data consistency
  • Make data retrieval easier
  • Keep your database well-structured and efficient

Summary

  • Relationships connect data across tables.
  • They are built using Primary Keys and Foreign Keys.
  • Three main types:
    • One-to-One (1:1)
    • One-to-Many (1:N)
    • Many-to-Many (M:N)
  • Many-to-Many relationships need an extra “junction” table.

In short: Relationships make your database come alive — they connect scattered data into meaningful information.
