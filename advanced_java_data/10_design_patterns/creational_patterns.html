🏗️ Creational Patterns

🎯 Goal

Creational design patterns deal with object creation. They help you create objects in a flexible, controlled way instead of scattering new everywhere.

🔑 Why Creational Patterns?

  • Hide complex construction logic
  • Improve testability (e.g., easier to mock)
  • Reduce coupling between client code and concrete classes
  • Support different creation strategies (lazy, pooled, cached, etc.)

🧱 Singleton Pattern

Ensures there is only one instance of a class, and it's globally accessible.

Use Cases

  • Configuration manager
  • Logger
  • Connection pool (though frameworks usually handle this)
public class AppConfig {

    private static final AppConfig INSTANCE = new AppConfig();

    private AppConfig() {
    }

    public static AppConfig getInstance() {
        return INSTANCE;
    }
}

In Spring, you rarely implement Singletons manually—Spring beans are singleton-scoped by default.

🧪 Factory Method Pattern

Defines an interface for creating objects, but lets subclasses decide which class to instantiate.

Example

public interface Notification {
    void send(String message);
}

public class EmailNotification implements Notification {
    public void send(String message) { /* ... */ }
}

public class SmsNotification implements Notification {
    public void send(String message) { /* ... */ }
}

public class NotificationFactory {
    public static Notification create(String type) {
        return switch (type) {
            case "email" -> new EmailNotification();
            case "sms" -> new SmsNotification();
            default -> throw new IllegalArgumentException("Unknown type" + type);
        };
    }
}

Client code depends on the Notification interface, not concrete classes.

🏭 Abstract Factory Pattern

Provides an interface for creating families of related objects without specifying their concrete classes.

Example: creating UI components for different platforms (Windows, Mac, Web) with consistent look and feel.

🧱 Builder Pattern

Separates object construction from its representation. Great for building complex objects with many optional parameters.

public class User {
    private final String username;
    private final String email;
    private final String phone;

    private User(Builder builder) {
        this.username = builder.username;
        this.email = builder.email;
        this.phone = builder.phone;
    }

    public static class Builder {
        private String username;
        private String email;
        private String phone;

        public Builder username(String username) {
            this.username = username;
            return this;
        }

        public Builder email(String email) {
            this.email = email;
            return this;
        }

        public Builder phone(String phone) {
            this.phone = phone;
            return this;
        }

        public User build() {
            return new User(this);
        }
    }
}

// Usage
User user = new User.Builder()
        .username("alice")
        .email("alice@example.com")
        .build();

📦 Prototype Pattern

Creates new objects by cloning existing instances.

  • Useful when object creation is expensive
  • Defines a clone() operation to copy objects

🔁 Object Pool Pattern (Bonus)

Maintains a pool of reusable objects instead of creating/destroying them frequently. Common for:

  • Database connections
  • Thread pools

🎯 Summary

  • Creational patterns control how objects are created
  • They help reduce coupling and support flexible construction
  • In enterprise Java, frameworks like Spring embody many of these patterns for you (Singleton, Factory, Builder via builders APIs, etc.)
