📡 Service Communication

🎯 How Do Microservices Talk to Each Other?

In a microservices system, services must collaborate to fulfill a request. Communication can be:

  • 📨 Synchronous – Request/response (e.g., HTTP/REST)
  • 📬 Asynchronous – Messaging (e.g., Kafka, RabbitMQ)

🌐 Synchronous Communication (HTTP/REST)

  • Client calls API Gateway
  • Gateway routes to one or more backend services
  • Services call each other over HTTP (often using REST)

Pros

  • Simpler mental model (like calling a function)
  • Easy to debug with tools like Postman, curl
  • Good fit for request/response flows

Cons

  • Tight temporal coupling – caller waits for callee
  • Chain failures – one slow/failing service can slow down many others
  • Harder to handle spikes gracefully

📨 Asynchronous Communication (Messaging)

Services communicate via message brokers (Kafka, RabbitMQ, ActiveMQ, etc.).

  • Producer sends a message to a topic/queue
  • Consumer services process messages independently
  • Supports event-driven architecture

Pros

  • Loose coupling – services don't need to be up at the same time
  • Better resilience and elasticity
  • Natural fit for event-driven and async workflows

Cons

  • More complex debugging and tracing
  • Requires careful design for idempotency and ordering
  • Eventual consistency instead of immediate consistency

🧬 Common Communication Patterns

  • Request/Response – HTTP/REST, typical for queries
  • Publish/Subscribe – Producers publish events; multiple subscribers react
  • Event Sourcing – State changes captured as a sequence of events
  • Command and Event – Command to perform an action, events to describe results

🧱 Example: Order Placement Flow

  1. Client calls Order Service (HTTP)
  2. Order Service creates an order and publishes OrderCreated event
  3. Inventory Service consumes the event and reserves stock
  4. Payment Service consumes the event and charges the customer
  5. Notification Service sends confirmation emails/SMS

Here, services collaborate via asynchronous events, decoupling their lifecycles.

🔍 Service Discovery

In a dynamic environment (containers, autoscaling), service instances come and go. Hardcoding URLs is not feasible.

  • Service Registry – A directory of available services (Eureka, Consul, etc.)
  • Service Discovery – Clients or gateways query the registry to find service instances

🛡️ Resilience Patterns

Network calls can fail. Use resilience patterns:

  • Timeouts – Never wait forever
  • Retries – Retry transient failures, with backoff
  • Circuit Breaker – Stop calling a failing service for a while
  • Bulkheads – Isolate resources so one failing dependency doesn't consume all threads

🧰 Tools & Libraries

  • Spring WebClient / RestTemplate – For HTTP calls between services
  • Spring Cloud OpenFeign – Declarative HTTP clients
  • Spring Cloud Netflix / Spring Cloud LoadBalancer – Load balancing & discovery integration
  • Resilience4j – Circuit breakers, retries, rate limiting

🎯 Summary

  • Choose HTTP/REST for simple request/response use cases
  • Use messaging for decoupled, event-driven flows
  • Implement resilience patterns and service discovery for production-grade systems
