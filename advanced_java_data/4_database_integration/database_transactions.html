🔐 Database Transactions

🎯 What is a Transaction?

A transaction is a sequence of one or more database operations that are treated as a single logical unit of work. Either all operations succeed, or none of them are applied.

🏦 Bank Transfer Analogy

Transferring money between two accounts involves:

  1. 💸 Deduct ₹1000 from Account A
  2. 💰 Add ₹1000 to Account B

If step 1 succeeds but step 2 fails, your system must rollback so money doesn't disappear. That's exactly what transactions guarantee.

🧬 ACID Properties

Transactions in relational databases follow the ACID properties:

  • A – Atomicity: All or nothing
  • C – Consistency: Move from one valid state to another
  • I – Isolation: Concurrent transactions don't interfere incorrectly
  • D – Durability: Once committed, changes survive crashes

🧱 Basic Transaction Operations

-- SQL level
BEGIN;                -- Start transaction
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT;               -- Save all changes

-- If something went wrong
ROLLBACK;             -- Undo all changes

⚙️ Transactions in Spring

Spring provides declarative transaction management via the @Transactional annotation.

1️⃣ Enabling Transactions

@Configuration
@EnableTransactionManagement
public class PersistenceConfig {
    // DataSource, EntityManagerFactory, TransactionManager beans
}

In Spring Boot with Spring Data JPA, transaction management is usually auto-configured. You just use @Transactional.

2️⃣ Using @Transactional

@Service
public class BankService {

    private final AccountRepository accountRepository;

    public BankService(AccountRepository accountRepository) {
        this.accountRepository = accountRepository;
    }

    @Transactional
    public void transfer(Long fromId, Long toId, double amount) {
        Account from = accountRepository.findById(fromId)
            .orElseThrow(() -> new IllegalArgumentException("Sender not found"));
        Account to = accountRepository.findById(toId)
            .orElseThrow(() -> new IllegalArgumentException("Receiver not found"));

        if (from.getBalance() < amount) {
            throw new IllegalStateException("Insufficient balance");
        }

        from.setBalance(from.getBalance() - amount);
        to.setBalance(to.getBalance() + amount);

        // Changes are automatically detected and flushed at commit time.
        // If any exception occurs, the transaction is rolled back.
    }
}

🔁 Transaction Propagation

Propagation defines what happens when a transactional method is called inside another transaction.

  • REQUIRED (default) – Join existing transaction or create a new one
  • REQUIRES_NEW – Always start a new transaction, suspending existing
  • SUPPORTS – Use transaction if exists, or run non-transactional
  • MANDATORY – Must run within existing transaction
  • NEVER – Must not run in a transaction
  • NOT_SUPPORTED – Suspend any existing transaction
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logAuditEvent(AuditEvent event) {
    // This runs in its own independent transaction
    auditRepository.save(event);
}

🧱 Isolation Levels

Isolation level controls how concurrent transactions see each other's data.

  • READ_UNCOMMITTED – Can see uncommitted changes (dirty reads)
  • READ_COMMITTED – Can only see committed data (default in many DBs)
  • REPEATABLE_READ – Same query returns same rows within a transaction
  • SERIALIZABLE – Highest isolation, like running sequentially
@Transactional(isolation = Isolation.REPEATABLE_READ)
public Order getOrderWithStableView(Long orderId) {
    return orderRepository.findById(orderId)
        .orElseThrow(() -> new IllegalArgumentException("Order not found"));
}

🚨 Rollback Rules

By default in Spring:

  • 🔄 Transaction is rolled back on unchecked exceptions (RuntimeException, Error)
  • ✅ Transaction is not rolled back on checked exceptions (like IOException) unless configured
@Transactional(rollbackFor = Exception.class)
public void processOrder(Long orderId) throws Exception {
    // Any exception (checked or unchecked) triggers rollback
}

🔍 Common Transaction Pitfalls

1️⃣ @Transactional on Private / Internal Methods

Spring creates proxies around beans to apply transactions. Calls must go through the proxy for @Transactional to work.

  • ❌ Won't work: Private methods, calls within same class
  • ✅ Will work: Public methods called from another bean

2️⃣ Long-Running Transactions

Avoid keeping transactions open for a long time:

  • ❌ Doing remote calls or heavy computations inside a transaction
  • ✅ Keep transactions short and focused on database work

3️⃣ Mixing Multiple DataSources

Distributed transactions across multiple databases require special configuration (JTA, XA transactions) and are more complex.

🧪 Example: Order Placement with Transaction

@Service
public class CheckoutService {

    private final OrderRepository orderRepository;
    private final PaymentService paymentService;
    private final InventoryService inventoryService;

    public CheckoutService(OrderRepository orderRepository,
                           PaymentService paymentService,
                           InventoryService inventoryService) {
        this.orderRepository = orderRepository;
        this.paymentService = paymentService;
        this.inventoryService = inventoryService;
    }

    @Transactional
    public Order checkout(Cart cart) {
        // 1. Reserve inventory
        inventoryService.reserveItems(cart.getItems());

        // 2. Create order
        Order order = createOrderFromCart(cart);
        Order savedOrder = orderRepository.save(order);

        // 3. Process payment
        paymentService.charge(savedOrder);

        // If payment fails, inventory reservation and order creation are rolled back
        return savedOrder;
    }
}

🎯 Key Takeaway

Transactions protect your data from corruption in the presence of failures and concurrency. In Spring, @Transactional and proper configuration of propagation, isolation, and rollback rules give you powerful control over how and when data changes are committed. Understanding transactions is essential for building reliable enterprise applications.
