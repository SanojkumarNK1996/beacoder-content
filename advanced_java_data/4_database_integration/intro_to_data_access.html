🔌 Introduction to Data Access in Java

🎯 Why Data Access Matters

Most real-world applications are data-driven — they read, store, update, and delete data. Data might be stored in:

  • 📊 Relational Databases (MySQL, PostgreSQL, Oracle)
  • 📁 Files (CSV, JSON, XML)
  • 🧠 In-memory caches (Redis, in-memory maps)

Data Access is the layer that connects your Java code to these data stores in a clean, reusable, and secure way.

🏦 Bank Analogy

Think of your application as a bank branch and the database as the central vault:

  • 👨‍💼 Tellers = Service layer
  • 🚪 Secure tunnel = Data access layer
  • 🏦 Vault = Database

The customer (user) never directly touches the vault. They interact through well-defined, secure processes. The same applies to data access.

🧱 Data Access Layer in a Layered Architecture

Client (Browser / Mobile)
        ↓
Controller Layer (Web / REST)
        ↓
Service Layer (Business Logic)
        ↓
Data Access Layer (Repositories / DAOs)
        ↓
Database (MySQL, PostgreSQL, etc.)
  • 🔹 Controller: Handles HTTP/web requests
  • 🔹 Service: Contains business rules
  • 🔹 Data Access: Talks to the database

Separating these layers makes your application easier to test, maintain, and evolve.

⚖️ JDBC vs ORM vs Spring Data

1️⃣ JDBC (Java Database Connectivity)

JDBC is the low-level, standard Java API for interacting with relational databases.

  • ✅ Advantages: Full control, standard API, no magic
  • ❌ Disadvantages: Lots of boilerplate code, manual mapping, error-prone
// Classic JDBC example
String sql = "SELECT id, name, email FROM users WHERE id = ?";

try (Connection connection = dataSource.getConnection();
     PreparedStatement statement = connection.prepareStatement(sql)) {

    statement.setLong(1, 1L);
    ResultSet rs = statement.executeQuery();

    if (rs.next()) {
        Long id = rs.getLong("id");
        String name = rs.getString("name");
        String email = rs.getString("email");
        // Map to User object manually
    }
} catch (SQLException e) {
    e.printStackTrace();
}

2️⃣ ORM (Object-Relational Mapping)

ORM frameworks like Hibernate and JPA map Java objects to database tables:

  • 🧱 Class = Table
  • 📌 Field = Column
  • 📄 Object instance = Row

Benefits:

  • ✅ Less SQL boilerplate
  • ✅ Type-safe queries (with JPA Criteria / query methods)
  • ✅ Works across different databases (portable)
@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String email;

    // getters/setters
}

3️⃣ Spring Data (JPA, JDBC, etc.)

Spring Data is a high-level abstraction built on top of JPA/JDBC that:

  • 🧾 Generates queries from method names
  • 📦 Provides ready-made repository interfaces
  • 🚀 Minimizes boilerplate to just interfaces
public interface UserRepository extends JpaRepository<User, Long> {
    
    // Spring Data derives query from method name
    List<User> findByEmail(String email);
    List<User> findByNameContainingIgnoreCase(String name);
}

🏗️ Typical Data Access Patterns

🧩 DAO / Repository Pattern

The DAO (Data Access Object) or Repository pattern encapsulates all database operations for a specific entity.

@Controller
@Service
@Repository (DAO)
Database

Example manual repository (without Spring Data JPA):

@Repository
public class UserRepository {

    private final JdbcTemplate jdbcTemplate;

    public UserRepository(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    public List<User> findAll() {
        String sql = "SELECT id, name, email FROM users";
        return jdbcTemplate.query(sql, new UserRowMapper());
    }

    public User findById(Long id) {
        String sql = "SELECT id, name, email FROM users WHERE id = ?";
        return jdbcTemplate.queryForObject(sql, new UserRowMapper(), id);
    }
}

🔐 Key Concerns in Data Access

  • 🔒 Security: Prevent SQL injection, protect credentials
  • ⚡ Performance: Use indexes, optimize queries, avoid N+1 problems
  • 🔁 Transactions: Ensure data consistency across multiple operations
  • 🧪 Testability: Mock repositories/services for unit tests

🛡️ Preventing SQL Injection

Never concatenate user input directly into SQL strings. Use prepared statements or frameworks like JdbcTemplate / JPA.

// ❌ Dangerous
String sql = "SELECT * FROM users WHERE email = '" + userInput + "'";

// ✅ Safe with placeholders
String sql = "SELECT * FROM users WHERE email = ?";
jdbcTemplate.queryForObject(sql, rowMapper, userInput);

🧩 Where Spring Fits In

Spring provides multiple options for data access:

  • 🧪 Spring JDBC – Thin layer over JDBC, less boilerplate
  • 📚 Spring Data JPA – High-level abstraction for JPA
  • 📦 Spring Data JDBC – Simpler than JPA, no lazy loading

In this topic, you will:

  1. Understand Spring JDBC Template for direct SQL with less boilerplate
  2. Learn JPA for ORM-based data modeling
  3. Use Spring Data JPA for repository-based development
  4. Manage transactions using Spring's @Transactional support

🎯 Key Takeaway

Data access is the bridge between your Java application and the database. Spring provides multiple abstraction levels so you can choose between full control (JDBC) and high productivity (Spring Data JPA). Understanding all three layers (JDBC, JPA, Spring Data) makes you a strong enterprise Java developer.
