🌐 REST Architecture Principles

🎯 What is REST?

REST (Representational State Transfer) is an architectural style for designing networked applications. In practice, REST is used to build web APIs that communicate over HTTP.

  • 🌍 Resources are things (users, orders, products)
  • 🔗 Each resource is identified by a URL
  • 📨 Clients interact using standard HTTP methods (GET, POST, PUT, DELETE, ...)
  • 📦 Data is typically exchanged as JSON

📚 Library Analogy

Imagine a digital library:

  • 📖 Books = Resources
  • 🔢 Book ID = Resource identifier (/books/10)
  • 👩‍💻 Librarian = API server
  • 🧾 Requests = HTTP requests

You “request” information about a book, “create” a new book, “update” or “delete” it using standard operations.

📦 Core Concepts

1️⃣ Resources and URIs

In REST, everything important is modeled as a resource.

  • /users – collection of users
  • /users/1 – a single user with ID 1
  • /users/1/orders – orders for user 1

Good RESTful URLs:

  • Use nouns, not verbs: /orders, /products
  • Use path for hierarchy: /users/1/orders/42
  • Use query parameters for filtering/sorting: /products?category=books&sort=price

2️⃣ HTTP Methods

Method Meaning Example
GET Read a resource GET /users/1
POST Create a new resource POST /users
PUT Update/replace a resource PUT /users/1
PATCH Partially update a resource PATCH /users/1
DELETE Delete a resource DELETE /users/1

3️⃣ HTTP Status Codes

  • 2xx – Success: 200 OK, 201 Created, 204 No Content
  • 4xx – Client Error: 400 Bad Request, 401 Unauthorized, 404 Not Found
  • 5xx – Server Error: 500 Internal Server Error, 503 Service Unavailable
// Example responses
200 OK
201 Created
404 Not Found
400 Bad Request

🧬 REST Constraints

REST defines a set of architectural constraints. If an API follows these, it can be called "RESTful".

1️⃣ Client-Server

  • Client (frontend/mobile) and server (backend API) are separate
  • They communicate via HTTP
  • Allows independent evolution of client and server

2️⃣ Stateless

  • Each HTTP request contains all the information needed
  • Server does not store client session between requests (no hidden state)
  • Makes scalability and reliability better

Example: Authentication done via JWT token sent with every request, not a server-side session.

3️⃣ Cacheable

  • Responses should define whether they can be cached
  • GET responses are often cacheable
  • Proper caching improves performance and reduces load

4️⃣ Uniform Interface

  • Consistent way of interacting with resources
  • Standard methods, URIs, and representations (JSON)
  • Makes APIs easier to understand and integrate

5️⃣ Layered System

  • API can be deployed with layers (gateway, load balancer, microservices)
  • Client doesn't need to know if it talks to a proxy, gateway, or actual service

📄 JSON Representations

Resources are typically represented in JSON.

GET /api/users/1

{
  "id": 1,
  "name": "Alice",
  "email": "alice@example.com",
  "links": {
    "self": "/api/users/1",
    "orders": "/api/users/1/orders"
  }
}

🧱 Designing RESTful APIs

1️⃣ Resource-Oriented URLs

  • ✅ /api/users, /api/users/1, /api/users/1/orders
  • ❌ /api/getUser, /api/createNewUser

2️⃣ Use HTTP Methods Correctly

  • GET /api/users – list users
  • GET /api/users/1 – get user 1
  • POST /api/users – create user
  • PUT /api/users/1 – replace user 1
  • PATCH /api/users/1 – partial update
  • DELETE /api/users/1 – delete user 1

3️⃣ Use Proper Status Codes

// Create user (success)
POST /api/users
201 Created
Location: /api/users/10

// Get unknown user
GET /api/users/999
404 Not Found

🧪 Example: Simple REST API Design

Resource: Book

GET    /api/books          - List all books
GET    /api/books/{id}     - Get book by ID
POST   /api/books          - Create new book
PUT    /api/books/{id}     - Replace existing book
PATCH  /api/books/{id}     - Partially update book
DELETE /api/books/{id}     - Delete book

GET    /api/books?author=John&page=0&size=10

🔐 REST and Security

Common patterns:

  • 🔑 Token-based authentication (JWT, OAuth2)
  • 🔒 HTTPS for encryption
  • 🧾 Role-based access control (RBAC)

🎯 Summary

  • REST is an architectural style for building web APIs
  • It focuses on resources, HTTP methods, and stateless communication
  • Good REST APIs use clean URLs, proper status codes, and JSON representations
