🚨 Exception Handling in REST APIs

🎯 Goal

Learn how to handle errors in a clean, consistent, and user-friendly way using Spring's exception handling features.

❌ What Happens Without Proper Handling?

If you don't handle exceptions:

  • Clients see HTML error pages instead of JSON
  • Error messages may expose internal details
  • Different endpoints return inconsistent error formats

For REST APIs, you want structured JSON error responses.

🧱 Basic Local Exception Handling

@GetMapping("/orders/{id}")
public Order getOrder(@PathVariable Long id) {
    try {
        return orderService.getOrderById(id);
    } catch (OrderNotFoundException ex) {
        throw new ResponseStatusException(HttpStatus.NOT_FOUND, ex.getMessage());
    }
}

This works, but scattering try/catch blocks everywhere is not clean. Better: a global exception handler.

🌍 Global Exception Handling with @ControllerAdvice

1️⃣ Define an Error Response Model

public class ApiError {

    private int status;
    private String error;
    private String message;
    private String path;
    private LocalDateTime timestamp = LocalDateTime.now();

    // constructors, getters, setters
}

2️⃣ Create a Global Exception Handler

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiError> handleResourceNotFound(ResourceNotFoundException ex,
                                                            HttpServletRequest request) {
        ApiError error = new ApiError();
        error.setStatus(HttpStatus.NOT_FOUND.value());
        error.setError("Not Found");
        error.setMessage(ex.getMessage());
        error.setPath(request.getRequestURI());

        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiError> handleValidationException(MethodArgumentNotValidException ex,
                                                               HttpServletRequest request) {
        String message = ex.getBindingResult().getFieldErrors().stream()
                .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
                .collect(Collectors.joining(", "));

        ApiError error = new ApiError();
        error.setStatus(HttpStatus.BAD_REQUEST.value());
        error.setError("Validation Failed");
        error.setMessage(message);
        error.setPath(request.getRequestURI());

        return ResponseEntity.badRequest().body(error);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiError> handleGenericException(Exception ex, HttpServletRequest request) {
        ApiError error = new ApiError();
        error.setStatus(HttpStatus.INTERNAL_SERVER_ERROR.value());
        error.setError("Internal Server Error");
        error.setMessage("An unexpected error occurred");
        error.setPath(request.getRequestURI());

        // Log the complete exception for debugging
        ex.printStackTrace();

        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}

🧪 Custom Domain Exceptions

@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}

// Usage in service layer
public User getUserById(Long id) {
    return userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User not found with id: " + id));
}

@ResponseStatus on the exception class automatically sets the HTTP status if thrown.

🧱 Handling Validation Errors

When using @Valid on request bodies, validation failures throw MethodArgumentNotValidException. Our global handler converts them into a clean JSON response.

@PostMapping("/users")
public ResponseEntity<User> createUser(@Valid @RequestBody CreateUserRequest request) {
    User user = userService.createUser(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(user);
}

// If validation fails, client might see:
{
  "status": 400,
  "error": "Validation Failed",
  "message": "email: must be a well-formed email address, password: Password must be at least 8 characters",
  "path": "/api/users",
  "timestamp": "2026-03-19T10:20:45.567"
}

🧩 Best Practices

  • Use structured error responses with consistent format
  • Never expose stack traces or internal details to clients
  • Log full exceptions on the server side
  • Map domain errors (e.g., "insufficient balance") to proper HTTP codes (e.g., 400)

🎯 Summary

  • Use @RestControllerAdvice and @ExceptionHandler for global error handling
  • Return consistent JSON error structures with meaningful messages
  • Combine with Bean Validation for robust input validation
