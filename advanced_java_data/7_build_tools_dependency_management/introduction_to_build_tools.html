🏗️ Introduction to Build Tools

🎯 Why Do We Need Build Tools?

As Java projects grow, manual compilation and packaging become painful:

  • Compiling dozens or hundreds of .java files
  • Managing library jars and versions
  • Running tests, generating reports
  • Packaging apps into JAR/WAR files

Build tools automate these tasks so you can focus on writing code.

🏭 Factory Analogy

Think of your project as a product:

  • 🔩 Source files = Raw materials
  • ⚙️ Build tool = Assembly line
  • 📦 JAR/WAR = Finished product

Build tools define and automate the steps to go from raw materials to final product.

🔧 Common Java Build Tools

  • 🧱 Ant – Older, XML-based, very flexible but verbose
  • 📦 Maven – Convention-over-configuration, dependency management via POM
  • ⚡ Gradle – Modern, Groovy/Kotlin DSL, powerful and flexible

In modern Java and Spring projects, Maven and Gradle are the most popular.

📦 What Do Build Tools Handle?

  • 🔨 Compilation – Compile Java source to bytecode
  • 🧪 Testing – Run unit/integration tests
  • 📦 Packaging – Create JAR/WAR artifacts
  • 📥 Dependency Management – Download and resolve libraries from repositories
  • 🚀 Deployment – Publish artifacts, run apps, build Docker images (via plugins)

🌐 Repositories

Build tools download dependencies from artifact repositories:

  • 🌍 Maven Central (https://repo.maven.apache.org/maven2)
  • 📦 JCenter (historically popular, now read-only)
  • 🏢 Private repositories like Nexus, Artifactory

🧱 Project Structure Conventions

Both Maven and Gradle typically follow a standard structure:

my-project/
├── src/
│   ├── main/
│   │   ├── java/          # Application source code
│   │   └── resources/     # Config files, templates, static assets
│   └── test/
│       ├── java/          # Test source code
│       └── resources/     # Test resources
├── target/ or build/      # Compiled output
├── pom.xml or build.gradle
└── README.md

Following conventions means tools & frameworks (like Spring Boot) can auto-detect your code and resources.

⚙️ Build Lifecycle (High Level)

Build tools define a lifecycle: ordered steps that run in sequence.

  • 🧹 clean – Remove previous build output
  • 🔨 compile – Compile source code
  • 🧪 test – Run tests
  • 📦 package – Create JAR/WAR
  • 🚀 install – Install artifact to local repository
  • ☁️ deploy – Deploy artifact to remote repository

🧪 Running Common Tasks

With Maven

# Clean and compile
mvn clean compile

# Run tests
mvn test

# Package as JAR/WAR
mvn package

# Install to local repository (~/.m2/repository)
mvn install

With Gradle

# Using Gradle wrapper (recommended)
./gradlew clean build

# Run tests
./gradlew test

# Build JAR
./gradlew jar

📦 Dependency Coordinates

Libraries are identified by their coordinates:

  • groupId – Organization (e.g., org.springframework.boot)
  • artifactId – Library name (e.g., spring-boot-starter-web)
  • version – Version number (e.g., 3.2.0)
# Maven example
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <version>3.2.0</version>
</dependency>

# Gradle example
dependencies {
  implementation "org.springframework.boot:spring-boot-starter-web:3.2.0"
}

🎯 Summary

  • Build tools automate compilation, testing, packaging, and dependency management
  • Maven and Gradle are the main tools for modern Java/Spring projects
  • They use standard project structures and lifecycles to simplify builds
