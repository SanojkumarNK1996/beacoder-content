💉 Dependency Injection

🎯 What is Dependency Injection?

Dependency Injection (DI) is a technique for implementing IoC where dependencies are provided (injected) to a class rather than the class creating them itself.

💉 Dependency Injection

A design pattern where objects receive their dependencies from external sources rather than creating them internally.

It's HOW we implement the WHAT of IoC

🏥 Medical Analogy: Getting Treatment

🏥 Without DI (Self-Treatment)

Patient handles everything:

  • 🔬 Diagnose your own illness
  • 💊 Research and buy medicine
  • 💉 Give yourself injections
  • 📋 Monitor your own recovery

Risky, error-prone, requires extensive knowledge!

Safer, reliable, specialized care provided when needed!

🔧 Types of Dependency Injection

1️⃣ Constructor Injection (Recommended)

Dependencies injected through class constructor

@Service
public class UserService {
    private final UserRepository userRepository;
    private final EmailService emailService;
    
    // Constructor injection - Spring calls this constructor
    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
    
    public User createUser(User user) {
        User savedUser = userRepository.save(user);
        emailService.sendWelcomeEmail(savedUser.getEmail());
        return savedUser;
    }
}

✅ Pros:

  • 🔒 Immutable: Dependencies can't change after creation
  • 🧪 Testable: Easy to pass mocks in tests
  • ✅ Required Dependencies: Clear what's needed
  • 🛡️ Null Safety: No null dependencies

❌ Cons:

  • 📝 More code if many dependencies
  • 🔄 Circular dependencies harder to solve

2️⃣ Setter Injection

Dependencies injected through setter methods

@Service
public class UserService {
    private UserRepository userRepository;
    private EmailService emailService;
    
    // Setter injection
    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
    
    @Autowired
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }
    
    public User createUser(User user) {
        User savedUser = userRepository.save(user);
        emailService.sendWelcomeEmail(savedUser.getEmail());
        return savedUser;
    }
}

✅ Pros:

  • 🔄 Optional Dependencies: Can have default values
  • 📝 Readable: Clear what each dependency does
  • 🔧 Reconfigurable: Can change dependencies at runtime

❌ Cons:

  • ⚠️ Null Risk: Dependencies might not be set
  • 🔄 Mutable: Dependencies can change unexpectedly
  • 🧪 Testing: Need to call setters in tests

3️⃣ Field Injection

Dependencies injected directly into fields

@Service
public class UserService {
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    private EmailService emailService;
    
    public User createUser(User user) {
        User savedUser = userRepository.save(user);
        emailService.sendWelcomeEmail(savedUser.getEmail());
        return savedUser;
    }
}

✅ Pros:

  • 📝 Less Code: Very concise
  • 🔧 Easy to Add: Just add @Autowired

❌ Cons:

  • 🧪 Hard to Test: Can't inject mocks easily
  • 🔍 Hidden Dependencies: Not obvious what's needed
  • 🏗️ Framework Coupling: Tied to Spring annotations
  • ⚠️ Null Risk: Dependencies might not be injected

🎯 DI Best Practices

1️⃣ Prefer Constructor Injection

// ✅ Good - Constructor injection
@Service
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    
    public OrderService(PaymentService paymentService, 
                       InventoryService inventoryService) {
        this.paymentService = paymentService;
        this.inventoryService = inventoryService;
    }
}

2️⃣ Use Interfaces for Dependencies

// ✅ Good - Depend on interface
@Service
public class NotificationService {
    private final MessageSender messageSender; // Interface
    
    public NotificationService(MessageSender messageSender) {
        this.messageSender = messageSender;
    }
}

// ✅ Multiple implementations possible
@Service
public class EmailMessageSender implements MessageSender { }

@Service  
public class SMSMessageSender implements MessageSender { }

3️⃣ Avoid Circular Dependencies

// ❌ Bad - Circular dependency
@Service
public class UserService {
    @Autowired
    private OrderService orderService; // UserService → OrderService
}

@Service  
public class OrderService {
    @Autowired
    private UserService userService;   // OrderService → UserService
}

// ✅ Good - Break the cycle
@Service
public class UserService {
    // Remove direct dependency on OrderService
}

@Service
public class OrderService {
    private final UserService userService;
    // Keep only one direction
}

4️⃣ Make Dependencies Final

// ✅ Good - Immutable dependencies
@Service
public class UserService {
    private final UserRepository userRepository; // final = immutable
    private final EmailService emailService;    // final = immutable
    
    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
}

🧪 Testing with Dependency Injection

✅ Easy Testing with Constructor Injection

// Production code
@Service
public class UserService {
    private final UserRepository userRepository;
    private final EmailService emailService;
    
    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
    
    public User createUser(String email) {
        User user = new User(email);
        User savedUser = userRepository.save(user);
        emailService.sendWelcomeEmail(email);
        return savedUser;
    }
}

// Test code
class UserServiceTest {
    @Test
    void shouldCreateUserAndSendEmail() {
        // Create mocks
        UserRepository mockRepository = mock(UserRepository.class);
        EmailService mockEmailService = mock(EmailService.class);
        
        // Inject mocks via constructor
        UserService userService = new UserService(mockRepository, mockEmailService);
        
        // Setup mock behavior
        User savedUser = new User("test@example.com");
        when(mockRepository.save(any(User.class))).thenReturn(savedUser);
        
        // Test
        User result = userService.createUser("test@example.com");
        
        // Verify
        assertThat(result.getEmail()).isEqualTo("test@example.com");
        verify(mockRepository).save(any(User.class));
        verify(mockEmailService).sendWelcomeEmail("test@example.com");
    }
}

❌ Harder Testing with Field Injection

// Production code with field injection
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository; // Hard to inject mocks
    
    @Autowired
    private EmailService emailService;     // Hard to inject mocks
    
    public User createUser(String email) {
        User user = new User(email);
        User savedUser = userRepository.save(user);
        emailService.sendWelcomeEmail(email);
        return savedUser;
    }
}

// Test code - more complex
@ExtendWith(SpringExtension.class)
class UserServiceTest {
    @InjectMocks
    private UserService userService;
    
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @Test
    void shouldCreateUserAndSendEmail() {
        // Need Spring test infrastructure
        // Dependencies injected via reflection
        
        User savedUser = new User("test@example.com");
        when(userRepository.save(any(User.class))).thenReturn(savedUser);
        
        User result = userService.createUser("test@example.com");
        
        assertThat(result.getEmail()).isEqualTo("test@example.com");
        verify(userRepository).save(any(User.class));
        verify(emailService).sendWelcomeEmail("test@example.com");
    }
}

⚠️ Common DI Problems & Solutions

🔄 Problem: Circular Dependencies

Symptom: BeanCurrentlyInCreationException

// ❌ Problem
@Service
public class A {
    @Autowired private B b;
}

@Service  
public class B {
    @Autowired private A a; // Creates cycle: A → B → A
}

// ✅ Solutions
// 1. Use @Lazy
@Service
public class A {
    @Autowired @Lazy private B b; // Delays B creation
}

// 2. Refactor design - extract common functionality
@Service
public class CommonService { }

@Service
public class A {
    @Autowired private CommonService common;
}

@Service
public class B {
    @Autowired private CommonService common;
}

🔍 Problem: No Qualifying Bean Found

Symptom: NoSuchBeanDefinitionException

// ❌ Problem - Multiple implementations
public interface PaymentService { }

@Service
public class CreditCardPayment implements PaymentService { }

@Service
public class PayPalPayment implements PaymentService { }

@Service
public class OrderService {
    @Autowired
    private PaymentService paymentService; // Which one?
}

// ✅ Solutions
// 1. Use @Qualifier
@Service
public class OrderService {
    @Autowired
    @Qualifier("creditCardPayment")  
    private PaymentService paymentService;
}

// 2. Use @Primary
@Service
@Primary  // This one is preferred
public class CreditCardPayment implements PaymentService { }

// 3. Inject specific implementation
@Service
public class OrderService {
    @Autowired
    private CreditCardPayment paymentService; // Specific type
}

🛠️ Advanced DI Features

🎯 Conditional Bean Creation

@Service
@ConditionalOnProperty(name = "payment.provider", havingValue = "stripe")
public class StripePaymentService implements PaymentService {
    // Only created if payment.provider=stripe in properties
}

@Service
@ConditionalOnProperty(name = "payment.provider", havingValue = "paypal")  
public class PayPalPaymentService implements PaymentService {
    // Only created if payment.provider=paypal in properties
}

📋 Bean Scopes

@Service
@Scope("singleton")    // Default - one instance per container
public class SingletonService { }

@Service
@Scope("prototype")    // New instance every time injected
public class PrototypeService { }

@Service
@Scope("request")      // One instance per HTTP request (web apps)
public class RequestScopedService { }

@Service
@Scope("session")      // One instance per HTTP session (web apps)
public class SessionScopedService { }

🎯 Key Takeaway

Dependency Injection is the implementation mechanism for IoC. It transforms tightly coupled code into flexible, testable, and maintainable applications.

💡 Golden Rule: Prefer constructor injection with final fields for mandatory dependencies. Use setter injection only for optional dependencies.
