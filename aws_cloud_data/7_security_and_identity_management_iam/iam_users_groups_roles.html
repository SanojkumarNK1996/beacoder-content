👥 IAM Users, Groups, and Roles

👤 IAM Users

Represents a person or service with long-term credentials (username/password for console, access key/secret for API). Best practice: one user per person, never share credentials.

👥 IAM Groups

A collection of users that share the same permissions. Attach policies to groups, not individual users. Examples: Developers group, Admins group, ReadOnly group.

🎭 IAM Roles

An identity with permissions that can be assumed by AWS services, users, or external identities (no permanent credentials). This is the preferred way to grant access.

  • EC2 instance role: Allows EC2 to call S3, DynamoDB without storing credentials on the instance
  • Lambda execution role: Permissions for Lambda to access other AWS services
  • Cross-account role: Allow another AWS account to access your resources

🎯 Summary

Use Groups to manage users at scale. Use Roles to grant permissions to AWS services and applications — never put credentials in your code.
